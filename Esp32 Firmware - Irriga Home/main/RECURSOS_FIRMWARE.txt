RECURSOS DO FIRMWARE - IrrigaHome (ESP32)
Ultima atualizacao: 2026-06-16

=========================================================
1) VISAO GERAL
=========================================================
Firmware modular para ESP32 (dual-core) com arquitetura orientada a gerenciadores.
- Core 0: MQTT, Firebase, WebServer
- Core 1: loop principal, sensores, controle, display
Operacao local independente de nuvem; persistencia via NVS (parametros/agendas) e
LittleFS (historico de eventos).

Arquivos-fonte:
  main.ino                   Orquestrador principal e logica de seguranca
  config.h                   Credenciais, pinos, topicos e constantes globais
  wifi_manager.{h,cpp}       Wi-Fi com backoff exponencial e NTP
  mqtt_manager.{h,cpp}       MQTT/TLS, telemetria delta, integracao Firebase
  sensor_manager.{h,cpp}     DHT22, umidade do solo (ADC), nivel de agua
  actuator_manager.{h,cpp}   Rele da bomba, modos, agendamentos (NVS)
  flow_meter_manager.{h,cpp} Fluxometro (ISR), calibracao linear por minimos quadrados
  irrigation_event_manager.{h,cpp}  Historico JSONL em LittleFS, fila MQTT
  display_manager.{h,cpp}    Dashboard TFT ST7789 240x320, flags de sujeira
  web_server_manager.{h,cpp} Servidor HTTP local de manutencao (porta 80)
  firmware_logger.h          Macros de log formatado com nivel e timestamp
  TFT_eSPI_Config.h          Configuracao minima do driver TFT_eSPI

3) LOOP PRINCIPAL (main.ino)
=========================================================
Ciclo de controle: 2 segundos (CONTROL_INTERVAL_MS = 2000)

Inicializacao (setup()):
  - Watchdog de tarefa (TWDT): timeout 30s, idle_core_mask=0
  - Todos os managers inicializados
  - Wi-Fi e NTP iniciados
  - Display com splash

Loop principal (loop()):
  1. Monitoramento de seguranca — fluxo e nivel de agua
  2. Verificacao de timeout de duracao ativa
  3. Irrigacao automatica por sensor de solo
  4. Irrigacao por agendamento
  5. Reset do watchdog (uma vez por ciclo)

Constantes do loop:
  CONTROL_INTERVAL_MS           2000  ms
  SOLO_SECO_PERSISTENCIA_MS    60000  ms  (solo deve ficar seco 1 min antes do auto)
  FLOW_FLOW_WARNING_TIMEOUT_MS  5000  ms  (aviso se sem vazao apos 5s)
  FLOW_NO_DETECTION_TIMEOUT_MS  6000  ms  (parada de seguranca apos 6s sem fluxo)
  WATCHDOG_TIMEOUT_SECONDS         8  s   (WDT do Core 1)

Funcao stopIrrigation(IrrigationStopReason reason):
  - Centraliza desligamento fisico (pino + LED)
  - Limpa activeUntil
  - Registra evento via IrrigationEventManager

Logica de irrigacao automatica (modoAuto=true):
  Requer: nivel de agua HIGH, umidade < limiar, cooldown esgotado,
          solo seco por 60s, solo nao acima de 75% (bloqueio)
  Acao:   atuador.ligar() + irrigationEventManager.onIrrigationStart(TRIGGER_AUTOMATIC)

Logica de agendamento (independe do modoAuto):
  Requer: relogio sincronizado, nivel de agua HIGH, bomba desligada, cooldown esgotado
  Usa:    atuador.checkDueSchedule() com janela de graca de 90 segundos
  Cooldown separado do automatico (lastScheduleTrigger)

4) CONECTIVIDADE (wifi_manager)
=========================================================
Classe: WiFiManagerCustom

Metodos:
  connect()      Inicia Wi-Fi assincronamente (STA mode, nao bloqueante)
  loop()         Mantem conexao com backoff exponencial + sincroniza NTP
  isClockReady() True somente apos NTP sincronizado e timestamp valido

Comportamento:
  - WiFi.setSleep(false) + esp_wifi_set_ps(WIFI_PS_NONE) para eliminar latencia
  - Backoff exponencial: 5s -> 10s -> 20s ... ate max 60s (reconexao e NTP)
  - Guard de timestamp: se < 1.000.000.000 (ano 2001) trata como invalido
  - Sem bloqueio do loop principal

5) MQTT E FIREBASE (mqtt_manager)
=========================================================
Classe: MqttManager
Dependencias injetadas: IrrigationEventManager (referencia)
Protocolo: MQTT sobre TLS (WiFiClientSecure + PubSubClient)

Metodos principais:
  begin(eventMgr)           Inicializa client, injeta dependencia
  loop()                    Mantem conexao, despacha fila firebase
  isConnected()             True se client.connected()
  publishIrrigationEvent()  Publica evento MQTT
  sendIrrigationEventToFirebase(...)  Enfileira para firebaseTask (Core 0)
  requestStopIrrigation(reason)  Sinal de parada para o loop principal
  isStopRequested() / consumeStopRequest()

Topicos:
  Assinar  commands: {"action":"irrigate","duration":30}
                     {"action":"setMode","mode":"auto"}
                     {"action":"setConfig","duration":20,"threshold":50,"cooldown":120}
                     {"action":"addSchedule","id":"s1","ativo":true,
                      "diasMask":127,"hour":6,"minute":0,"durationSeconds":30}
  Publicar telemetry: {"solo":45,"temp":28.5,"umidAr":72,"nivelAgua":true,"bomba":false}
  Publicar status:    {"modo":"manual","duracao":10,"threshold":40,"agendamentos":2}

Telemetria delta:
  - Compara valores novos com cache (lastSolo, lastTemp, lastUmidAr, lastNivelAgua, lastBomba)
  - Publica somente se algum valor mudou E janela de 3s (TELEMETRY_STABLE_WINDOW_MS) esgotada
  - Reduz carga no broker e banda

Conexao MQTT assincrona:
  - mqttConnectTask (Core 0, 32KB stack) executa TLS handshake separadamente
  - Flag mqttJustConnected sinaliza Core 1 apos conexao
  - Core 1 assina topicos e envia mensagens enfileiradas

Firebase (firebaseTask, Core 0):
  - FreeRTOS queue (firebaseQueue) recebe eventos de Core 1
  - Envia HTTP GET para Cloud Function com parametros de consulta
  - Apos HTTP 200: chama g_eventMgr->removeFromHistory(eventId)
  - Sem registro no TWDT (TLS pode bloquear > 8s)

Parser de comandos MQTT:
  - Funcoes proprias: findFirstKey, parseAction
  - Tolera payloads malformados
  - Valida action, duration, mode e campos de agendamento

NVS namespace "sensoroffset":

  off_solo    int
  off_umid_ar float

Mapeamento do solo: map(adc, 4095, 0, 0, 100) -> porcentagem calibrada

8) FLUXOMETRO (flow_meter_manager)
=========================================================
Classe: FlowMeterManager

Metodos:
  begin()                           Init GPIO (GPIO 23), carrega calibracao da NVS
  startMeasurement()                Vincula ISR (FALLING), zera contador
  stopMeasurement()                 Desvincula ISR, registra resultado
  getTotalPulses()                  Leitura atomica via mutex
  getTotalVolumeLiters()            Aplica calibracao
  getAvgFlowRateLpm()               Litros por minuto durante a medicao
  hasFlowDetected()                 True se pulsos > 0
  setPulsesPerLiter(float)          Atualizacao atomica
  setVolumeCalibration(scale,off)   Calibracao simples (multiplicativa + offset)
  addCalibrationSample(pulses,ml)   Adiciona amostra para calibracao linear
  computeLinearCalibration()        Regressao linear por minimos quadrados
  clearCalibrationSamples()
  getLinearCalibrationCoefficients(a,b)
  addCalibrationSampleFromCurrentMeasurement(realVolumeMl)
  setCalibrationVolumeCorrectionFactor(factor)
  loadCalibration() / saveCalibration()  NVS namespace "flowcal"

Dois modos de calibracao:
  Simples:  volume_mL = (pulses / pulsesPerLiter) * 1000 * scale + offset
  Linear:   volume_mL = a * pulsos + b  (minimos quadrados, ate 64 amostras)

ISR:
  _isr() em IRAM_ATTR, incrementa _pulseCount atomicamente
  Protecao de acesso: portENTER_CRITICAL

NVS namespace "flowcal":
  scale   float
  offset  float
  a, b    float (coeficientes lineares)

Constantes:
  FLOW_PULSES_PER_LITER   260000
  PUMP_NOMINAL_FLOW_ML_PER_MIN  600
  MAX_CAL_SAMPLES         64
  LEGACY_FLOW_VOLUME_SCALE  1.1489 (detectado -> migra automaticamente)

9) EVENTOS E HISTORICO (irrigation_event_manager)
=========================================================
Classe: IrrigationEventManager

Enum IrrigationStopReason:
  STOP_REASON_COMPLETED=0  Timeout normal
  STOP_REASON_MANUAL=1     Parada pelo usuario
  STOP_REASON_NO_WATER=2   Caixa vazia
  STOP_REASON_NO_FLOW=3    Sem vazao detectada

Struct IrrigationEvent (fila RAM, max 8):
  eventId, startAt, endAt, durationSec
  trigger, stopReason, sentToMqtt
  totalPulses, totalVolumeLiters, avgFlowRateLpm
  totalVolumeMl, accountingVolumeMl, nominalFlowRateMlPerMin
  flowDetected, flowStatus ("OK" | "ZERO_FLOW" | "ANOMALY")

Struct IrrigationHistoryEntry (LittleFS + buffer RAM, max 20):
  Mesmos campos do IrrigationEvent + sentToFirebase
  isSuccess(): stopReason == STOP_REASON_COMPLETED

Metodos:
  begin()                        Init LittleFS, carrega estado ativo e historico
  onIrrigationStart(trigger)     Registra inicio, gera eventId, persiste estado
  onIrrigationEnd(reason)        Calcula volume, salva em fila + LittleFS
  update()                       Detecta estado orfao apos reboot (intervalo 5s)
  isIrrigationActive()
  getActiveIrrigationSeconds()
  getPendingEventCount()
  getNextPendingEvent(outEvent)
  markEventSent(eventId)
  removeFromHistory(eventId)     Thread-safe (mutex), chamado por firebaseTask
  getHistoryCount()
  getHistoryEvent(index, outEvent)
  formatIso8601(timestamp, buf)  Utilitario estatico
  generateEventId(buf)           timestamp + sufixo aleatorio

Volume sem fluxometro:
  volume_mL = durationSec * (72.0 / 6.0) = 12 mL/s (baseado no fluxo nominal)

Arquivos LittleFS:
  /irrg_state.json    Estado de irrigacao ativa (recuperacao pos-reboot)
    {"isActive":true,"startAt":1708612345,"trigger":2,"eventId":"1708612345-abc1"}

  /irrg_history.jsonl  Uma linha JSON por evento
    {"eventId":"...","startAt":...,"endAt":...,"durationSec":30,"trigger":2,
     "stopReason":0,"totalVolumeMl":360.0,"accountingVolumeMl":300.0,
     "nominalFlowRateMlPerMin":600.0,"success":true,"sentToFirebase":false}

Thread safety:
  SemaphoreHandle_t _historyMutex (binario FreeRTOS)
  Core 0 (firebaseTask) e Core 1 (loop) compartilham historyBuffer

Deduplicacao:
  sentIds[40][24] — cache volatil em RAM
  isAlreadySent(eventId) / markIdAsSent(eventId)
  Limpo ao reiniciar (sem persistencia)

Recuperacao pos-reboot:
  update() checa a cada 5s se activeState.isActive=true mas bomba fisicamente desligada
  Se detectado: chama onIrrigationEnd(STOP_REASON_COMPLETED)

11) WEB SERVER DE MANUTENCAO (web_server_manager)
=========================================================
Classe: WebServerManager
Porta: HTTP 80, executado no Core 0

Dependencias injetadas: SensorManager, ActuatorManager, FlowMeterManager

Endpoints:
  GET  /               Pagina HTML de manutencao (SPA em PROGMEM)
  GET  /api/data        JSON com leituras ao vivo (polling de 3s pelo browser)
  POST /api/auth        Login (campo: password)
  POST /api/logout      Encerra sessao
  POST /api/measure     Inicia/para medicao do fluxometro
  POST /api/config      Atualiza offsets de sensor e calibracao de fluxo
  POST /api/resetSchedules  Limpa todos os agendamentos da NVS

Pagina HTML:
  - SPA em PROGMEM (sem dependencias externas)
  - Exibe leituras em tempo real
  - Formulario de offsets (temperatura, solo, umidade do ar) e calibracao de fluxo
  - Autenticacao com limite de 3 tentativas
  - Layout responsivo (mobile-friendly)

Config pending (thread-safe):
  POST /api/config grava em estrutura _pending (Core 0)
  applyPendingConfig() aplicado no loop principal (Core 1) — sem conflito de acesso

Autenticacao:
  Flag in-memory _authValid
  Limite de 3 tentativas antes de bloquear sessao

Execucao:
  _task() como tarefa FreeRTOS no Core 0
  Nao bloqueia Core 1 (loop de controle)

16) PADROES ARQUITETURAIS
=========================================================
- Injecao de dependencia:  gerenciadores injetados como referencia (sem extern cego)
- Nao bloqueante:          Wi-Fi/MQTT/Firebase em tasks separadas com backoff
- Thread safety:           semaforo FreeRTOS no historico; flags volatile entre cores
- Watchdog:                TWDT Core 1, timeout 30s (cobre pior caso TLS)
- Telemetria delta:        publica so se valor mudou + janela de estabilidade de 3s
- Batch NVS:               dirty flag + flush unico evita escritas frequentes na flash
- Recuperacao pos-reboot:  estado ativo persistido em LittleFS, verificado a cada 5s

Fim.
